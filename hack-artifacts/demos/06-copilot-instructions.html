📜 Copilot Instructions & Prompt Files

Teach your team how to embed coding standards into Copilot and build reusable prompt templates that supercharge productivity across the entire team.

🗂️ File Hierarchy Overview

These are the key files Copilot reads for project context and reusable prompts.

📁 demo-app/
  ├── 📁 .github/
  │   ├── 📄 copilot-instructions.md ← Project-level rules
  │   └── 📁 prompts/ ← Reusable templates
  │       ├── 📝 add-fighter.md
  │       └── 📝 add-test-suite.md ← Created live in demo
  ├── 📁 src/
  ├── 📄 package.json
  └── 📄 tsconfig.json

🎬 Setup

Have these ready before you begin:

  • VS Code open with the demo-app/ project folder
  • .github/copilot-instructions.md visible in the Explorer sidebar
  • GitHub Copilot Chat panel open (Agent mode enabled)
  • No other chat conversations open (start fresh)
  • Font size increased for audience visibility (Cmd/Ctrl + =)
1

Step 1 📄 Show the Instructions File

Open the file and walk through each rule so the audience understands what Copilot will enforce.

💬 Say:

"This file tells Copilot HOW your team writes code. Every response respects these rules."

⌨️ Do:

Open .github/copilot-instructions.md in the editor. Scroll through the file slowly — point out each rule as you go.

Walk through each rule in the file. Highlight the patterns the team has standardized:

# Key rules to point out:
- JSDoc comments on every exported function
- Use `const` over `let` wherever possible
- Consistent error handling pattern (try/catch with typed errors)
- Structured API response format { success, data, error }
👀 What They See:

The copilot-instructions.md file with clear, readable rules about JSDoc comments, const usage, error handling patterns, and structured API response format.

💡 Teaching Point:

This is like a .eslintrc for AI — organizational coding standards that Copilot follows automatically. Every suggestion, every generation, every refactor will respect these rules.

2

Step 2 🧪 Demo the Impact

Prove the instructions work by generating code and inspecting the output for compliance.

💬 Say:

"Let's prove it works. Watch how Copilot follows these rules."

⌨️ Type in Chat (Agent mode):
Create a new analytics endpoint at /api/analytics that returns battle statistics

Wait for Copilot to generate the code. Then walk through the output with the audience, checking for compliance:

# ✅ Check the generated code for:
- JSDoc comments on functions? ✓
- const declarations (not let)?  ✓
- try/catch error handling?      ✓
- { success, data, error } shape? ✓
👀 What They See:

Generated code that matches the project's style perfectly — JSDoc on every function, const declarations throughout, consistent error handling, and the structured response format. It looks like a senior developer on your team wrote it.

💡 Teaching Point:

Instructions = org-wide coding standards enforced automatically. No more "please follow our conventions" in code review. The AI already knows your team's style.

3

Step 3 📋 Show Prompt Files

Transition from project-level rules to reusable prompt templates.

💬 Say:

"Prompt files are reusable templates. Instead of typing the same complex prompt every time, save it as a file."

⌨️ Do:

Open .github/prompts/add-fighter.md in the editor. Show its structure to the audience.

Point out how the prompt file defines a repeatable pattern — all the context and requirements baked in.

⌨️ Type in Chat:
Using #file:.github/prompts/add-fighter.md, add a new 🌙 Moon fighter
👀 What They See:

Copilot creates a complete fighter following the prompt file template — consistent structure every time. The Moon fighter has all the same properties, formatting, and patterns as existing fighters.

💡 Teaching Point:

Prompt files = team productivity multiplier. New team member? Give them the prompt library. They're immediately productive, generating code that matches your team's patterns from day one.

4

Step 4 ✍️ Create a New Prompt File Live

Build a prompt file from scratch so the audience sees how easy it is to create their own.

💬 Say:

"Let's create one together — this is something you'll do on your own team."

⌨️ Create file .github/prompts/add-test-suite.md with:
# Add Test Suite

Create a comprehensive test suite for the specified service.

## Requirements
- Use vitest with describe/it blocks
- Include setup and teardown
- Test happy paths and error cases
- Mock external dependencies
- Aim for >80% coverage

## Structure
- Group tests by method
- Use descriptive test names
- Include edge cases

Now use the prompt file you just created:

⌨️ Type in Chat:
Using #file:.github/prompts/add-test-suite.md, create tests for the battle service
👀 What They See:

Consistent, well-structured tests generated from the template — vitest describe/it blocks, setup/teardown, mocked dependencies, happy paths and error cases. Every team member will produce the same quality of tests.

💡 Teaching Point:

Teams can share prompt libraries — put them in your repo, version them with git, iterate together. It's collaborative prompt engineering. Your best prompts become team assets, not tribal knowledge.
